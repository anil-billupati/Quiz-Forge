## **ABOUT THIS DOCUMENT** 

This PRD specifies the ContestForge Core Contest Engine. It defines the central concept of the Contest Mode (Standard, Speed, Elimination), how the two contest structures (Normal and Grouped) apply that mode, and the supporting Execution, Scoring, Wildcard, Leaderboard, and Elimination engines. It is the authoritative reference for engineering, design, and QA. 

What changed in v2.3:  Section 9 now states durability as requirements (no answer loss, at-most-once scoring, authoritative timestamp, recoverability) rather than prescribing an implementation. Earlier, v2.2 reduced wildcards to three (Fifty-Fifty, Second Chance, Skip), removed Block Opponent and Extra Time, and made Skip award full points (floor in Speed mode). 

Earlier in v2.0:  Contest Mode became the central concept and applies to BOTH Normal and Grouped quizzes. Each Mode now has an explicit definition and a fixed scoring model. A glossary, core concept model, and clarified configuration scope have been added to remove ambiguity.

## **1. Core Concepts & Glossary** 

## **1.1  The Core Model in One Picture** 

ContestForge has one organising idea: every quiz runs in a Mode, and the Mode determines how questions are timed, scored, and how participants are removed. A contest's Structure (Normal or Grouped) only decides WHERE that configuration lives — for the whole quiz, or per group. 

## `CONTEST` 

```
  ├─ Structure = Normal   →  one Configuration Block applies to the whole quiz
  └─ Structure = Grouped  →  each Group has its own Configuration Block
```

```
CONFIGURATION BLOCK (per quiz, or per group)
  ├─ Mode            = Standard | Speed | Elimination
  ├─ Scoring Model   = determined by Mode (see Section 5)
  ├─ Timing          = question duration, interval, explanation, leaderboard
  ├─ Wildcards       = which wildcards are enabled + limits
  └─ Elimination     = rules (only when Mode = Elimination)
```

Key consequence: Standard, Speed, and Elimination modes are available to BOTH Normal and Grouped quizzes. A Normal Quiz can be a Speed quiz; a Grouped Quiz can mix a Standard group with an Elimination group. 

## **1.2  Glossary** 

|**Term**|**Definition**|
|---|---|
|**Contest**|A complete quiz event with a lifecycle, participants, and results.|
|**Structure**|How a contest is organised: Normal (flat) or Grouped (questions split into<br>groups).|
|**Configuration Block**|The set of mode, timing, scoring, wildcard, and elimination settings. Lives at<br>quiz level (Normal) or group level (Grouped).|
|**Mode**|The behaviour profile of a quiz/group: Standard, Speed, or Elimination.|
|**Group**|A named sub-section of a Grouped quiz with its own Configuration Block.|
|**Participant**|A registered user competing in the contest.|
|**Moderator**|An admin who can control reveal timing and progression during a live contest.|
|**Wildcard**|A power-up a participant can activate for a temporary advantage.|
|**Checkpoint**|A point at which elimination rules are evaluated.|
|**Survivor**|A participant not yet eliminated in an Elimination-mode quiz/group.|

## **2. Contest Model** 

## **2.1  Contest Structures** 

A contest uses exactly one structure, chosen at creation and locked once the contest is published. The structure does not limit which Modes are available — it only determines where the Configuration Block lives. 

|**Structure**|**Where config lives**|**Use For**|
|---|---|---|
|**Normal Quiz**|One Configuration Block for the entire quiz;<br>all questions share it.|Certifications, assessments,<br>single-domain or short<br>competitions.|
|**Grouped Quiz**|One Configuration Block per group; groups<br>run in sequence.|Multi-round events, hiring<br>challenges, mixed-mode<br>competitions.|



Clarification (v2.0): In v1.0, Modes were described as group-only. That was incorrect. A Normal Quiz also has a Mode. A Normal Speed quiz applies one Speed configuration to every question; a Grouped quiz simply repeats the same kind of configuration per group. 

## **2.2  Grouped Quiz Example** 

A Grouped Quiz lets each group run a different Mode. Groups are optional — a Grouped contest may contain anywhere from one group upward. 

```
Contest:  Fresher Hiring Challenge   (Structure = Grouped)
  Group 1  Aptitude       Mode = Standard      30s/Q   Fixed scoring
  Group 2  Java           Mode = Speed         20s/Q   Time-based scoring
  Group 3  System Design  Mode = Elimination   90s/Q   Top 10 advance
```

## **2.3  Configuration Block Fields** 

Whether at quiz level (Normal) or group level (Grouped), a Configuration Block contains the same fields: 

|**Field**|**Allowed Values / Notes**|
|---|---|
|**Mode**|Standard | Speed | Elimination (see Section 3)|
|**Scoring Model**|Determined by Mode — not independently set (see Section 5)|
|**Question Duration**|5 – 300 seconds per question|
|**Question Interval**|0 – 60 seconds between questions|
|**Explanation Duration**|0 – 60 seconds the correct-answer explanation is shown|

## **2.4  Contest Lifecycle** 

All contests, regardless of structure or mode, follow the same linear lifecycle. No stage may be skipped. 

|**Stage**|**Description**|
|---|---|
|**Draft**|Being configured. Not visible to participants. All fields editable.|
|**Published**|Metadata publicly visible. Configuration becomes locked next stage.|
|**Registration Open**|Participants may register. Configuration is now locked.|
|**Registration Closed**|Registration window ended. Participant list finalised.|
|**Scheduled**|A start time (`scheduled_start_at`) is set and the contest is armed to go Live automatically.|
|**Live**|Actively running. Question delivery and scoring in progress.|
|**Completed**|All questions done or end time reached. Final scores calculated.|
|**Archived**|Read-only historical record. Results and leaderboards preserved.|

**Automatic start:** the transition **Scheduled → Live is automatic** — the platform
arms the contest at `scheduled_start_at` and goes Live when that time is reached, with
no human action required. All other transitions are explicit operator actions. A
moderator may still control reveal/progression once Live (see §4.3).



## **3. Contest Modes** 

## **3.1  Overview** 

The Mode is the single most important setting in a Configuration Block. It defines how questions behave, which scoring model is used, and whether participants can be eliminated. There are exactly three modes, and each is available to both Normal and Grouped quizzes. 

## **3.2  Mode Definitions** 

|**Mode**|**What it does**|**Scoring**|**Elimination**|
|---|---|---|---|
|**Standard**|Each question runs for a fixed duration. Everyone<br>answers; points depend only on correctness. The<br>default, fairest mode.|Fixed|No|
|**Speed**|Identical flow to Standard, but faster answers earn<br>more points. Rewards quick, confident responses.|Time-Based|No|
|**Elimination**|Runs like Standard, but participants who fail a<br>configured rule are removed and cannot continue.|Fixed|Yes|

Rule: Mode determines the scoring model. Standard and Elimination use Fixed scoring; Speed uses Time-Based scoring. Scoring cannot be set independently of Mode. 

## **3.3  Standard Mode** 

- Every participant sees each question for the same fixed duration. 

- Correct answers earn a flat point value; wrong answers earn zero (no negative marking). 

- No participant is removed during the quiz — all play to the end. 

- Best for certifications and assessments where fairness and equal opportunity matter most. 

## **3.4  Speed Mode** 

- Same question-by-question flow as Standard mode. 

- Points decrease the longer a participant takes to answer correctly (Time-Based scoring, Section 5.3). 

- A correct answer always earns at least the configured minimum (floor) score. 

- No elimination. Best for fast-paced competitions and tie-sensitive rankings. 

## **3.5  Elimination Mode** 

- Runs like Standard mode, with Fixed scoring, but adds elimination rules and checkpoints. 

- At each checkpoint, participants who fail the configured rule are removed and cannot answer further questions. 

- A Survivor Leaderboard (Section 7) tracks remaining participants from the first checkpoint onward. 

- Best for knockout rounds and narrowing a large field to a small set of finalists. 

## **4. Execution Engine** 

## **4.1  Purpose** 

The Execution Engine is the authoritative controller of question delivery, timers, submission windows, and progression during a Live contest. All timing is computed server-side; client clocks are used for display only and never accepted as authoritative. 

## **4.2  Execution Flow (per question)** 

|**Question Display**|**Timer Starts**|**Answer Submission**|**Evaluation**|
|---|---|---|---|
|||||
|**Explanation**|**Leaderboard**|**Interval**|**Next Question**|



This loop repeats until the quiz (or group) is exhausted or the end time is reached. The duration of each phase comes from the active Configuration Block.

## **4.3  Reveal Modes** 

Reveal Mode controls when a question becomes visible. It is independent of the Contest Mode and configured per quiz or group. 

|**Reveal Mode**|**Behaviour**|
|---|---|
|**Automatic**|Questions reveal on a fixed schedule with no human action.|
|**Moderator Controlled**|A moderator triggers each reveal manually; the interval timer pauses between<br>questions.|


## **4.4  Progression** 

In a Normal Quiz, progression is simply question-to-question. In a Grouped Quiz, the engine also advances between groups. A group ends — and the next begins — when one of the following occurs: 

- All questions in the current group are completed. 

- Elimination criteria are satisfied (only in Elimination-mode groups, see Section 8). 

- A moderator issues an override to advance. 

Between groups, the engine pauses to display the group leaderboard and announce any eliminations before the next group's Configuration Block takes effect. 

## **4.5  Non-Functional Requirements** 

- Server-side timer accuracy within ±50ms over a 5-minute session. 

- Answers submitted after the server-side close time are rejected, regardless of client clock. 

## **5. Scoring Engine** 

## **5.1  Overview** 

All scoring is performed server-side. The scoring model in use is dictated by the Mode of the active Configuration Block, not chosen separately. 

|**Mode**|**Scoring Model Applied**|
|---|---|
|**Standard**|Fixed Scoring (Section 5.2)|
|**Speed**|Time-Based Scoring (Section 5.3)|
|**Elimination**|Fixed Scoring (Section 5.2), plus elimination rules|

## **5.2  Fixed Scoring** 

Points depend only on correctness. Response time has no effect. Used by Standard and Elimination modes. 

|**Outcome**|**Points**|
|---|---|
|**Correct Answer**|+10 (configurable, default 10)|
|**Wrong Answer**|0 — no negative marking|
|**No Answer / Timeout**|0|
|**Second Chance (correct)**|Reduced rate, default 50% of full points|

Note (v2.3 decision): negative marking is **not supported**. A wrong answer always
scores 0 and a participant's cumulative score never falls below 0.



## **5.3  Time-Based Scoring** 

Points decrease as response time increases. Used by Speed mode. The bands below are the default and are configurable per Configuration Block. 

|**Time Range (s)**|**Points**|**Notes**|
|---|---|---|
|**0 – 5**|100|Maximum band|
|**6 – 10**|75|Fast response|
|**11 – 15**|50|Average response|
|**16 – 20**|25|Slow response|
|**20+**|10 (floor)|Minimum for a correct answer|
|**Timeout / No Answer**|0|No floor applies|



Band matching is deterministic: a band's range is its **upper-bound-inclusive** interval and the **first (fastest) matching band wins**. Each band is configured by its upper bound (`max_seconds`); a response is placed in the lowest band whose upper bound it does not exceed. Example: an answer at exactly 5.000 s scores 100; at 5.001 s it scores 75. Response time is measured server-side from the authoritative reveal instant (`QuestionWindow.revealed_at`) to server acceptance, not from per-client delivery. 

Administrators may instead replace the bands with a linear decay: Points = max(floor, maxPoints − elapsedSeconds × decayRate). Bands and decay are mutually exclusive per Configuration Block. 

## **5.4  Tie-Breaking** 

When scores are equal, rank is resolved by applying these criteria in order until the tie breaks. The logic is deterministic and logged for audit. 

1. Fastest total completion time (sum of per-question response times). When a participant uses Second Chance on a question, that question contributes the **second attempt's** response time (and the second attempt's time also drives Speed scoring for that question). 

2. Fewest wrong answers across the quiz. 

3. Earliest timestamp of the last correct submission. 

## **5.5  Group Score Rollup (Grouped Quiz only)** 

Group scores aggregate into a contest-level score using one configurable strategy: 

|**Strategy**|**Calculation**|
|---|---|
|**Sum (default)**|Contest score = sum of all group scores|
|**Weighted Sum**|Contest score = Σ (group score × group weight)|

## **6. Wildcard Engine** 

## **6.1  Overview** 

Wildcards are optional power-ups, disabled by default, that a participant may activate for a temporary advantage. They are configured per Configuration Block (per quiz for Normal, per group for Grouped) and are subject to eligibility. Each enabled wildcard may be used **at most once per participant for the entire contest** (a participant may, however, use more than one wildcard on the same question — there is no per-question limit). There is no cooldown and no per-group reset; once a wildcard is used anywhere in the contest it is spent. 

## **6.2  Supported Wildcards** 

|**Wildcard**|**Effect**|
|---|---|
|**Fifty-Fifty**|Removes two incorrect options from the current question. The correct answer is<br>always preserved. Cannot be used after an answer is selected.|
|**Second Chance**|After a wrong answer, allows one more attempt at the same question. Points on<br>the second attempt are reduced (default 50%).|
|**Skip Question**|Skips the current question — it is not shown or attempted — and awards full<br>points. In Fixed scoring (Standard / Elimination) the participant receives the full<br>correct-answer value; in Speed mode the participant receives the floor score<br>(Section 5.3).|



## **6.3  Configuration** 

|**Setting**|**Description**|
|---|---|
|**Enabled Wildcards**|Which of the three types are available in this Configuration Block|
|**Usage Limit**|Fixed: each enabled wildcard is usable **once per participant for the whole contest** (not configurable, not per-group)|
|**Eligibility**|Which participants may use it: `ALL` or `TOP_50_PERCENT` (top 50% by score). Top-50% is evaluated against the **last committed leaderboard at the start of the current question**|

Note (v2.3 decision): cooldown and per-group carry-over/reset have been removed. A wildcard is spent once used, anywhere in the contest.



## **6.4  Audit** 

Every activation is logged with participant ID, wildcard type, question number, timestamp, and outcome, and is included in post-contest result exports.

## **7. Leaderboard Engine** 

## **7.1  Overview** 

The Leaderboard Engine maintains rankings in near real-time. Rankings are computed server-side and pushed to clients, and are the authoritative source of rank at any moment. 

## **7.2  Views** 

|**View**|**Description**|**Available In**|
|---|---|---|
|**Contest Leaderboard**|Aggregate ranking across the whole contest.|All contests|
|**Group Leaderboard**|Ranking within one group; resets at group<br>start.|Grouped Quiz|
|**Survivor**<br>**Leaderboard**|Non-eliminated participants only; updates at<br>each checkpoint.|Elimination mode|



## **7.3  Ranking Criteria** 

The primary sort key is configurable per Configuration Block: 

|**Criterion**|**Sort**|**Tie-Break**|
|---|---|---|
|**Score Only**|Highest score first|Apply Section 5.4 sequence|
|**Score + Time**|Score, then shortest total time|Fewest wrong, then earliest last submission|
|**Accuracy**|Highest correct %|Score, then fastest time|

Accuracy is computed as correct answers ÷ **questions revealed so far** (not total contest questions and not only those attempted), so the percentage is comparable across participants at any point in the run. 



## **7.4  Update Frequency** 

- After every answer — freshest, highest load. For ≤500 participants. 

- After every question — recalculated when the timer closes. Default. 

- After every group — for large contests (>5,000 participants). 

Performance: updates pushed within 500ms at ≤5,000 concurrent users; within 2s at peak load (≤20,000). 

## **7.5  Tie Display & Visibility** 

|**Setting**|**Behaviour**|
|---|---|
|**Shared Rank (default)**|Tied participants share a rank; next rank is skipped (1, 1, 3).|
|**Fastest Participant**|Tie broken toward the lowest cumulative response time.|
|**Least Incorrect**|Tie broken toward fewer wrong answers.|
|**Visibility**|Always visible | Post-question only | Hidden | Masked (own rank only)|

## **8. Elimination Engine** 

## **8.1  Overview** 

The Elimination Engine removes underperforming participants. It is active only when the Configuration Block's Mode is Elimination. In a Grouped Quiz, only Elimination-mode groups are affected; other groups are untouched. 

## **8.2  Elimination Rules** 

Rules may be combined with AND/OR logic within one Configuration Block. 

|**Rule**|**Description**|
|---|---|
|**First Wrong Answer**|Eliminated on the first incorrect answer. For precision rounds.|
|**N Wrong Answers**|Eliminated after N wrong answers (default N = 3).|
|**Bottom X Percent**|At a checkpoint, the lowest-ranked X% of active participants are eliminated. If the X% cut-off falls inside a group of tied participants, **all tied participants in the cut are eliminated** (the cut never splits a tie).|
|**Minimum Score**<br>**Threshold**|Eliminated if a defined score is not reached by the checkpoint.|



## **8.3  Checkpoints** 

|**Checkpoint**|**Trigger**|
|---|---|
|**After Question**|Evaluated immediately after a configured question closes.|
|**After Group**|Evaluated after all questions in the group complete.|
|**Custom Milestone**|Evaluated at an admin-defined timestamp or event.|



## **8.4  Flow** 

|**Checkpoint Reached**|**Rule Evaluated**|**Eliminated Set Found**|**Notify**|
|---|---|---|---|
|||||
|**Survivor List Locked**|**Leaderboard Updated**|**Next Question / Group**|**(survivors only)**|



## **8.5  Post-Elimination Behaviour** 

- Eliminated participants are notified with their final rank and score. 

- They may be granted spectator (view-only) access to later groups. 

- Survivors carry accumulated scores forward unless a reset is configured. 

- The Survivor Leaderboard activates from the first checkpoint onward.

## **9. Failure Handling & Acceptance Criteria** 

## **9.1  Design Guarantees** 

The system must preserve contest integrity under partial failure: no submitted answer is ever lost, no participant is mis-scored due to a transient fault, and contest state is reconstructible from durable storage at any time. 

## **9.2  Failure Scenarios & Required Behaviour** 

|**Scenario**|**Required Behaviour**|
|---|---|
|**Participant Disconnect**|On reconnect, the participant's current contest state is restored and their<br>submission window is honoured as it stood at disconnect.|
|**Application / Server**<br>**Crash**|On restart, the contest resumes with no loss of recorded progress and no<br>duplicate scoring.|
|**Cache Failure**|Rankings remain correct and are recoverable from authoritative data; a<br>cache loss must not change any participant's score or rank.|
|**Persistence-Layer**<br>**Failure**|An accepted answer must never be lost; the system must hold and reconcile<br>it until it is durably recorded.|
|**Network Failure**|Transient network faults must not lose answers or advance the contest<br>incorrectly; affected participants are notified.|



## **9.3  Answer Durability Requirements** 

This section states the durability guarantees the platform must provide. It does not prescribe an implementation; engineering may meet these requirements with any suitable design. 

- Durability: once an answer is accepted, it is never lost — it survives any single component failure (server, cache, or persistence layer). 

- At-most-once scoring: each accepted answer is counted exactly once; retries or restarts must not double-count or drop a submission. 

- Authoritative timestamp: the time used for scoring is the moment the server first accepted the answer, regardless of any later retry or recovery delay. 

- Acknowledgement: a participant is told whether their answer was accepted, and an accepted answer remains accepted even if confirmation is delayed by a fault.

- Recoverability: contest state and recorded answers are reconstructible from authoritative storage at any point in the contest. 

## **9.4  Acceptance Criteria** 

|**Criterion**|**Pass Condition**|
|---|---|
|**No answer loss**|Zero answers lost across 10,000 submissions with a 1% induced<br>persistence-failure rate.|
|**Contest recovery**|Contest resumes within 30s of a simulated crash with state intact and no<br>double-scoring.|
|**Ranking recovery**|Rankings remain correct after a total loss of cached data.|
|**Consistent scoring**|Score totals match before failure and after recovery.|
|**Mode correctness**|Each mode applies its correct scoring model in 100% of regression<br>cases.|
|**Reconnection**|Reconnecting participant restored within 3s.|
